# 마이링크 (MyLink) - 제품 요구사항 정의서 (PRD)

## 1. 프로젝트 개요
- **프로젝트명**: 마이링크 (MyLink)
- **목표**: 흩어져 있는 다양한 링크(SNS, 블로그, 포트폴리오 등)를 하나의 페이지로 직관적으로 관리하고 공유할 수 있는 무료 멀티 링크 프로필 서비스를 구축합니다.
- **과금 정책**: 전면 무료 (수익화 모델 없음)

---

## 3. 핵심 시스템 디자인 (개발 지침)

### 3.1. 프로필 정보 분리 (URL 슬러그 vs 실제 노출 이름)
* **URL 슬러그용 ID (`displayName`)**: 사용자 프로필 웹페이지 링크의 끝단을 구성하는 라우팅 식별자입니다. 가입 시 사용한 **구글 지메일 계정의 앞부분**(예: `abc@gmail.com`의 `abc`)을 추출하여 초기값으로 자동 할당합니다.
* **화면 노출용 표시 이름 (`username`)**: 실제 랜딩 화면 상단 프로필 헤더에 굵게 표기되는 사용자의 실제 활동명 또는 본명입니다.
* 두 필드(`displayName`, `username`)를 명확하게 분리하여 사용자가 원하는 이름으로 자신을 표현하도록 시스템화합니다.

### 3.2. 이음새 없는 편집 UX (Inline Edit)
* 프로필의 텍스트(`username`, 소개글)와 개별 등록된 링크 컨텐츠들을 바꾸고자 할 때, 불편한 등록 폼 페이지나 팝업으로 이동하지 않고 리스트 문구를 직접 눌러 곧바로 텍스트 수정 상태로 연결되는 **인라인 편집 (Inline Edit)** 기능을 반영해야 합니다.

### 3.3. 파비콘(Favicon) 일관성 부여 (Google API)
* 개별 링크마다 아이콘 사진을 일일이 첨부하지 않고, 사용자가 입력한 URL의 공식 파비콘을 그대로 가져옵니다. 
* 방식을 일원화하기 위해 `https://s2.googleusercontent.com/s2/favicons?domain=웹사이트주소` 형태의 구글 제공 범용 API에 도메인을 삽입해 자동으로 파비콘을 노출합니다.

### 3.4. 데이터베이스 및 저장소 아키텍처
* **가입 연동 자동화**: Firebase Auth를 통해 승인 완료 시 제공되는 이메일 주소를 파싱하여 `users` 컬렉션의 `displayName` 필드에 값을 삽입하는 로직을 필수 구현합니다.
* **서브컬렉션(Subcollection) 구조**: 데이터를 가볍고 최적화된 상태로 불러오기 위해 사용자가 만들어내는 개개의 링크 묶음은 해당 사용자 정보 문서 하위에 속하는 서브 컬렉션(Subcollection) 형태로 분리 저장합니다.
